## Why

The project has no durable, reproducible view of how much code it carries or where
that code lives. A one-off audit found the raw line count (~229K LOC) is dominated by
*generated* LLVM IR snapshots and *tracking/reference* material, while the hand-authored
compiler and runtime — the code whose complexity actually matters — is a small fraction
(~8K LOC). That distinction is exactly what a lines-of-code total hides. We want a
catalogue that classifies code by component, role, and language; that is regenerated on
demand so it never goes stale; and whose classification policy is updated deliberately as
the architecture and file layout evolve.

## What Changes

- Add a **complexity catalogue tool**: a narrating engine that walks the git-tracked
  files, classifies each by component (path), role (authored / generated / vendored /
  reference / tracking / test / demo / spike), and language (extension), and reports
  line counts as a component×language×role catalogue.
- Add a persisted, human-readable **`docs/COMPLEXITY.md`** whose mechanical tables are
  regenerated by the tool (idempotently, between generated-section markers) and whose
  qualitative interpretation is authored/refreshed alongside it.
- Add a **`make catalogue`** target so the tables regenerate without an agent and can run
  in CI, and a **`complexity-catalogue` skill** that runs the engine and refreshes both
  the tables and the interpreted commentary in one step.
- Encode the **authored-vs-generated-vs-vendored classification as explicit rules** in the
  tool, so the catalogue stays honest when files move and the policy is versioned with the
  code it describes.
- The tool conforms to the existing `tooling-observability` conventions: narration on
  stderr, data on stdout, metrics reported, controllable via `EMIT_VERBOSITY`.

## Capabilities

### New Capabilities
- `complexity-catalogue`: The requirement that a reproducible catalogue tool classifies
  tracked code by component, role, and language; reports line counts; and regenerates a
  persisted `docs/COMPLEXITY.md` idempotently with an explicit, maintainable
  classification policy and generation provenance.

## Impact

- **New files**: `docs/COMPLEXITY.md`; a catalogue engine under `tools/` (e.g.
  `tools/complexity.sh`); a `.claude/skills/complexity-catalogue/` skill.
- **Modified files**: `Makefile` (new `catalogue` target); possibly `README.md`/`docs`
  index to point at the new document.
- **Dependencies**: none beyond the existing toolchain (`git`, `wc`, POSIX shell,
  `tools/log.sh`). No effect on the compiler, runtime, or any shipped artifact.
