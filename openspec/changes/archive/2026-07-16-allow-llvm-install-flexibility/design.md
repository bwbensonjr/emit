## Context

The toolchain location is hardcoded to macOS Homebrew (`/opt/homebrew/opt/llvm@22`, libgc under
`/opt/homebrew`) in four places, with the same five values duplicated:

| Location | Overridable today? | Notes |
|---|---|---|
| `Makefile:29-37` | No (`:=`, not `?=`) | builds the C++ ORC hosts; derives flags via `llvm-config` |
| `bin/scheme-compile:21-24` | Yes (`${VAR:-…}`) | Chez-free AOT driver, `$CC` link |
| `tools/regen.sh:30-35` | Yes (`${VAR:-…}`) | rebuilds committed IR |
| `src/compile.ss:45-46,157-159` | **No** (`define` constants) | Chez driver; the only `--backend jit/bitcode` impl |

`--backend jit` and `--backend bitcode` exist *only* in `src/compile.ss` (invoked as
`chez --libdirs src --script src/compile.ss … --backend jit`). That file also carries two real
portability defects beyond the paths: `gc-dylib` is `libgc.dylib` (macOS-only, breaks `lli -load=`
on Linux), and the AOT path plus the host-target-triple probe shell out to a **bare** `clang`
(`compile.ss:54`, `:174`) that need not be on PATH. On a stock Linux/apt box — LLVM 22 at
`/usr/lib/llvm-22` (`llvm-config-22`), libgc 8.2.6 via `pkg-config bdw-gc`, no `clang` on PATH —
nothing builds and no backend runs.

The system `clang` is already used by bare name for AOT while jit/bitcode use the pinned keg — an
existing asymmetry. Narration conventions are set by `docs/OUTPUT.md` + `tools/log.sh`
(`say`/`vsay`, stderr-only, `EMIT_VERBOSITY`); `src/compile.ss` mirrors them (`note`, `EMIT_VERBOSITY`).

## Goals / Non-Goals

**Goals:**
- Any install where `llvm-config` and libgc are discoverable builds and runs all backends with no
  file edits: Linux/apt, Intel/ARM Homebrew, Nix, custom.
- Environment overrides win, through one consistent interface across build + all drivers.
- Version becomes a documented soft floor (warn, never fail; no upper bound), not an exact pin.
- Detection is single-sourced; the 4× duplication is removed.
- Fix the `.dylib` JIT bug and the bare-`clang` AOT assumption.

**Non-Goals:**
- Changing the emitted IR dialect (`aot-codegen`'s "LLVM 22 syntax" is unchanged).
- Supporting compilers other than clang/LLVM, or GCs other than Boehm libgc.
- Rewriting the `spike/*/run.sh` experiments (non-shipping; out of scope).
- Adding a configure step or new build system — discovery stays a thin shell layer + Make fragment.

## Decisions

### D1. `llvm-config` + `pkg-config` as the discovery anchors
`llvm-config --bindir` yields the tool directory (→ `clang`, `clang++`, `lli`, `llvm-as`,
`llvm-link`); `--cxxflags`/`--ldflags --libs orcjit native --system-libs` already supply the ORC-host
build flags (unchanged formula). `pkg-config bdw-gc` yields libgc's include/lib dirs.
- **Why**: these are the canonical, layout-independent discovery tools and both are present on the
  target machine. `llvm-config` already gates the Makefile's flags today.
- **Alternative rejected**: PATH scan for version-suffixed tool names (`clang-22`, `llc-15`). Brittle,
  no single version source, and misses off-PATH kegs.

### D2. A single shared shell layer: `tools/llvm-env.sh`
One script does detection and exports both the shell-consumer names
(`CC CXX LLVM_CONFIG CXXFLAGS LDFLAGS GC_INC GC_LIB GC_DYLIB`) and `EMIT_LLVM_*` mirrors
(`EMIT_LLVM_BIN EMIT_GC_INC EMIT_GC_LIB EMIT_GC_DYLIB`). Consumers:
- `bin/scheme-compile`, `tools/regen.sh`, `test/self-host-fixpoint.sh`: `. tools/llvm-env.sh`
  (they already honor these var names; this just replaces the macOS defaults with discovered ones).
- `Makefile`: `-include build/llvm.mk`, generated by `tools/llvm-env.sh --print-make`; values stay
  overridable from env/command line.
- Narrates via `tools/log.sh` (`say`/`vsay`), stderr, honoring `EMIT_VERBOSITY`.
- **Why**: kills the 4× duplication with one detection implementation; keeps the Make world in Make
  and the shell world in shell without a second detector.
- **Alternative rejected**: `$(shell …)` per-variable in the Makefile — reintroduces detection logic
  into Make and diverges from the shell path.

### D3. `src/compile.ss`: env interface + self-probe fallback
The Chez driver cannot source bash. It reads `EMIT_LLVM_BIN` / `EMIT_GC_INC` / `EMIT_GC_LIB` /
`EMIT_GC_DYLIB` (via `getenv`, as it already does for `EMIT_VERBOSITY`/`SCHEMEC`); when unset it
self-probes by shelling `llvm-config --bindir` and `pkg-config` (it already shells out for the target
header at line 54). Precedence: env override → self-probe → last-resort legacy default.
- **Why**: honors the same override interface the shared layer exports (single interface, D2) while
  keeping `chez --script src/compile.ss … --backend jit` working with no launcher.
- Touch points: `gc-inc`/`gc-lib` (45-46), `llvm-bin` (157), `gc-dylib` (159), `require-llvm-tools`
  (163-170), AOT `link`/target probe (54, 174), and the modular jit/bitcode backends (~405-438,
  468-469) that reuse `tool`/`gc-dylib`.

### D4. Platform-correct shared-lib extension
Derive `.dylib` (Darwin) vs `.so` (else) from `uname -s` in the shell layer and from the platform in
`compile.ss` (or probe which `libgc.<ext>` exists under the lib dir). `gc-dylib` becomes
`<gc-lib>/libgc.<ext>`. Fixes the JIT backend on Linux.

### D5. Version floor, not pin
Parse `llvm-config --version` major. Below a documented floor (**19**, released Sept 2024;
overridable via `EMIT_LLVM_MIN`) → one `say` warning, then continue. No upper bound.
`require-llvm-tools` probes the discovered tools and, on miss, errors with a platform-neutral message
naming both `apt install llvm-<N>` and `brew install llvm@<N>` and the `EMIT_LLVM_BIN` override.
- **Why 19**: comfortably recent (a modern LLVM is assumed anyway); higher than a bare
  opaque-pointers floor and easy to raise/lower via `EMIT_LLVM_MIN`.
- **Alternative rejected**: hard-fail outside a range — contradicts the "any recent LLVM" goal and
  creates churn each LLVM release.

### D6. AOT clang: prefer system clang, fall back to discovered
AOT resolution order: `CC`/env override → `clang` on PATH → discovered `$LLVM_BIN/clang`. This
preserves the current, intentional "AOT uses the *system* clang" default on macOS (Apple clang links
cleanly against the system SDK, serving the small-clean-binaries goal) while still working with zero
setup on a Linux box that has a discoverable LLVM but no `clang` on PATH. Applies to
`compile.ss:54` (target-triple probe) and `:174` (link), and to `bin/scheme-compile`'s `CC` default.
- **Why not require `clang` on PATH**: it would save only two `compile.ss` edits (discovery already
  computes the bin dir), while forcing an extra `apt install clang` on an otherwise-complete install.
- **Why not always use the discovered clang**: that silently moves the macOS AOT default off Apple
  clang; the fallback keeps the existing default and only engages when no `clang` is on PATH.

## Risks / Trade-offs

- **macOS non-regression can't be exercised on the Linux dev box** → keep Homebrew prefixes
  (`/opt/homebrew/opt/llvm@*/bin`, `/usr/local/opt/llvm@*/bin`) in the search list and `.dylib` on
  Darwin; verify the branches by reading, and note the residual risk for a macOS reviewer.
- **`pkg-config` may lack a `bdw-gc` entry on some installs** → documented fallback to prefix-based
  `include`/`lib` (the current Homebrew paths) before failing.
- **Self-probe in `compile.ss` partially duplicates the shell detector** → keep the Scheme probe to
  the two minimal queries (`--bindir`, `pkg-config`), with the shared layer remaining the source of
  truth for anything richer; the env interface is the contract that keeps them aligned.
- **Floor choice (15) is a judgment call** → make it a documented, overridable constant so it is easy
  to raise if the emitted IR ever needs a newer consumer.

## Migration Plan

1. Add `tools/llvm-env.sh`; wire `Makefile`, `bin/scheme-compile`, `tools/regen.sh`, and the test
   scripts to it. 2. Rework `src/compile.ss` to the env+self-probe model and fix `.dylib`/bare-clang.
3. Update docs. No committed IR changes (`bootstrap/*.ll` are host-agnostic inputs); `make regen`
should still converge. Rollback is reverting the change — the committed IR and specs are untouched.
